# Helmsman — Role-Based Capability Gates & Approval Protocol
> Definitive spec. Applies to AWS today, extends to GCP, Kubernetes, Terraform, and everything else.

---

## The Three Roles

Every action Helmsman can take belongs to exactly one role. Roles stack — activating Commander also requires Operator to be active.

---

### 🔍 Observer
**Always active. Cannot be disabled.**

Any action that only reads, inspects, lists, monitors, or diagnoses. Zero risk. Zero approval needed.

Covers: list, describe, get, inspect, check, monitor, diagnose, search, fetch, show, view, status, logs, metrics, health, audit

Examples:
- List EC2 instances, S3 buckets, Lambda functions
- Read CloudWatch metrics and logs
- Describe security groups, IAM policies, RDS config
- Check Kubernetes pod status
- Read a file on a remote server
- View GitHub issues, PRs, workflow runs

---

### ⚙️ Operator
**Disabled by default. Activated explicitly. Expires after 30 minutes.**

Any action that creates, modifies, configures, deploys, starts, stops, scales, updates, or applies changes. These actions change infrastructure but are generally recoverable.

Covers: create, add, deploy, start, stop, restart, update, modify, configure, scale, apply, install, enable, disable, attach, detach, tag, upload, push, invoke

Examples:
- Create an EC2 instance, S3 bucket, Lambda function
- Update a security group rule
- Deploy a Kubernetes workload
- Push code to a repository
- Start or stop an EC2 instance
- Update a Lambda function's code
- Modify an RDS parameter group
- Write a config file on a server

---

### ⚡ Commander
**Disabled by default. Activated explicitly. Expires after 15 minutes.**
**Requires Operator to be active first.**
**Each action confirmed individually, one resource at a time.**

Any action that is irreversible or very difficult to recover from. These destroy data, remove infrastructure, or permanently alter state.

Covers: delete, remove, terminate, destroy, purge, wipe, drop, revoke, detach-and-delete, force-delete, empty

Examples:
- Terminate an EC2 instance
- Delete an S3 bucket and its contents
- Delete a Lambda function
- Remove an IAM role or policy
- Delete a Kubernetes namespace
- Destroy a Terraform-managed resource
- Overwrite a critical config file on a production server

## Activation Flow

### Step 1 — Agent detects a capability gap

User: "Create an EC2 instance called memora-prod"

Agent checks: is Operator active? No.

Agent responds:
```
To create resources, Operator access needs to be enabled first.

Operator access lets me create, modify, and configure infrastructure.
It expires automatically after 30 minutes.

To activate, type this exactly:
/activate operator R7K2MX
```

The ID is generated by the backend, stored with the pending activation record, and expires in 5 minutes if unused. The user must type the command exactly — no extra text, no punctuation, nothing else on the line.

---

### Step 2 — Backend intercepts, never hits the LLM

The `/activate operator R7K2MX` message is caught by the transport handler before it reaches the agent.

Backend logic:
1. Parse the command: role = `operator`, id = `R7K2MX`
2. Look up the ID in the activation store
3. Verify: correct user, correct chat, not expired, not already consumed
4. Mark Operator as active for this `userId + chatId`, set expiry to `now + 30min`
5. Mark the activation ID as consumed
6. Respond directly to the user (no LLM involved):

```
✅ Operator access activated

You can now create, modify, and configure infrastructure.
Active until: 14:32 UTC (30 minutes)

Resuming your request...
```

7. Re-trigger the original user request now that the role is active

---

### Step 3 — Agent proceeds, generates approval request

Now that Operator is active, the agent plans the EC2 creation and presents an approval request (see full approval format below).

## Approval Messages

Every Operator or Commander action requires an individual approval before execution. The approval message shows:

1. **What will happen** — plain English, no jargon, written for a non-technical reader
2. **The exact command** — for engineers who want to verify
3. **Metadata** — expiry time, risk level, resource being affected
4. **The confirmation instruction** — exactly what to type

---

### Operator Approval Format

```
⚙️ Operator Action — Confirmation Required

What this does:
  Creates a new virtual server named "memora-prod" in the US East (N. Virginia) 
  region. It will be a t3.medium instance (2 vCPUs, 4GB RAM) running Amazon 
  Linux 2023. This server will start billing immediately at ~$30/month.

Command that will run:
  aws ec2 run-instances \
    --image-id ami-0c55b159cbfafe1f0 \
    --instance-type t3.medium \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=memora-prod}]' \
    --metadata-options HttpTokens=required \
    --region us-east-1

Resource: EC2 Instance — memora-prod
Risk: Operator (reversible — instance can be stopped or terminated later)
Expires: This approval expires at 14:28 UTC (10 minutes)

To confirm, type:
/approve P4N7Q3
```

---

### Commander Approval Format

```
⚡ Commander Action — Destructive Confirmation Required

What this does:
  Permanently terminates the EC2 instance "web-prod-1". The server will be 
  immediately shut down and all locally stored data on its instance store 
  will be lost forever. EBS volumes attached with "delete on termination" 
  will also be permanently deleted. This cannot be undone.

  ⚠️  This instance has been running for 47 days and showed CPU activity 
  3 minutes ago. It may be serving live traffic.

Command that will run:
  aws ec2 terminate-instances \
    --instance-ids i-0524aa25c11382aa1 \
    --region us-east-1

Resource: i-0524aa25c11382aa1 (web-prod-1)
State: running | Uptime: 47 days | Last activity: 3 min ago
Risk: Commander (IRREVERSIBLE — cannot be undone)
Expires: This approval expires at 14:22 UTC (10 minutes)

To confirm termination, type:
/confirm i-0524aa25c11382aa1
```

This keeps transport parsing deterministic while still requiring explicit resource-level confirmation.

---

### Multiple Commander Actions — Always Split

If a task contains more than one Commander action, they are never batched. Each one is confirmed sequentially, and the second confirmation is only presented after the first executes successfully.

```
This task has 2 Commander actions. I'll handle them one at a time.

⚡ Commander Action 1 of 2 — Destructive Confirmation Required

What this does:
  Permanently terminates the EC2 instance "web-prod-1"...

[user confirms first]

✅ Instance i-0524aa25c11382aa1 terminated.

⚡ Commander Action 2 of 2 — Destructive Confirmation Required

What this does:
  Permanently deletes the S3 bucket "test-bucket-ikram-123" and all 847 
  objects inside it (2.3 GB). Once deleted, the bucket name may be claimed 
  by someone else. The data cannot be recovered unless you have backups elsewhere.

  Last accessed: 12 days ago. No lifecycle rules or replication configured.

Command that will run:
  aws s3 rb s3://test-bucket-ikram-123 --force --region us-east-1

[user confirms second]
```

There is no "approve all" for Commander actions. Ever.

## The Action Payload (Internal)

Every pending approval is stored as a typed record. The backend executes this when confirmation arrives.

```typescript
type PendingAction = {
  // Identity
  id: string                      // 6-char alphanumeric: "P4N7Q3"
  role: 'operator' | 'commander'
  userId: string
  chatId: string
  platform: 'telegram' | 'slack' | 'cli'

  // Timing
  createdAt: Date
  expiresAt: Date                 // operator: 10min, commander: 10min
  consumed: boolean

  // What the user sees
  summary: {
    plainEnglish: string          // non-technical explanation
    resourceName: string          // "web-prod-1"
    resourceId: string            // "i-0524aa25c11382aa1"
    resourceType: string          // "EC2 Instance"
    riskNote?: string             // "appears to be serving live traffic"
  }

  // Confirmation requirement
  confirmationMode: 'code' | 'resource_id'
  confirmationTarget: string      // for code: "P4N7Q3", for resource_id: "i-0524aa25c11382aa1"

  // What actually runs (never shown to user in raw form)
  execution: {
    tool: string                  // "shell_execute"
    params: Record<string, unknown>
  }

  // What happens after execution
  onSuccess: {
    responseMessage: string
    resumeWorkflowRunId?: string  // if part of a multi-step workflow
    resumeStepId?: string
  }
}
```

The `execution.params` — the raw tool call — is stored here but **never shown to the user in the approval message in isolation**. The user sees the plain-English summary and the command string formatted for readability. They do not see the internal tool name or parameter structure.

## Role State

```typescript
type RoleState = {
  userId: string
  chatId: string
  observer: { active: true }       // always true, immutable
  operator: {
    active: boolean
    activatedAt?: Date
    expiresAt?: Date
  }
  commander: {
    active: boolean
    activatedAt?: Date
    expiresAt?: Date
  }
}
```

Role state is checked at two points:
1. **Before planning** — if the plan requires a role that isn't active, pause and request activation before generating the plan
2. **Before execution** — re-verify at execution time (the window may have expired between planning and confirming)

## Context Enrichment Before Commander Confirmations

Before presenting a Commander approval, the agent automatically fetches context about the resource. The user should never be asked to confirm deletion of something without knowing what it actually is right now.

### EC2 termination — agent fetches:
- Instance name, type, state, region, uptime
- Recent CloudWatch CPU metrics (is it active?)
- Whether it's in a load balancer target group (is traffic hitting it?)
- EBS volumes attached with delete-on-termination

### S3 bucket deletion — agent fetches:
- Object count and total size
- Last modified date of most recent object
- Whether versioning, replication, or lifecycle rules are configured
- Whether any CloudFront distribution uses it as an origin

### Lambda deletion — agent fetches:
- Last invocation time and invocation count (last 7 days)
- Whether any event source mappings are attached
- Whether it's behind an API Gateway

### RDS deletion — agent fetches:
- Instance state, uptime, storage size
- Whether automated backups are enabled
- Whether deletion protection is on (and warn if being bypassed)

This context appears in the "What this does" section of the Commander confirmation and in the `riskNote` field if anything looks dangerous.

## Transport Intercept Logic

The transport handler checks incoming messages in this order before passing to the agent:

```typescript
async function handleIncoming(message: NormalizedMessage) {
  const text = message.text.trim()

  // 1. Activation commands
  if (/^\/activate (operator|commander) [A-Z0-9]{6}$/.test(text)) {
    return handleRoleActivation(message)
  }

  // 2. Operator approvals (code-based)
  if (/^\/approve [A-Z0-9]{6}$/.test(text)) {
    return handleOperatorApproval(message)
  }

  // 3. Commander confirmations (command-based)
  if (/^\/confirm\s+[^\s]+$/.test(text)) {
    return handleCommanderConfirmation(message)
  }

  // 4. Normal message — goes to the agent
  return agentService.handleMessage(message)
}
```

The regex patterns ensure accidental matches are impossible:
- Activation: must start with `/activate`, must be a valid role name, must be exactly 6 uppercase alphanumeric characters
- Approval: must start with `/approve`, must be exactly 6 uppercase alphanumeric characters
- Commander confirmation: must start with `/confirm` and include one token resource target

## Storage Requirement (Redis)

All role-gate records are written to Redis before prompting the user:

- Pending activation records (`/activate`) with 5-minute TTL
- Role state records (operator/commander active windows)
- Pending action records for `/approve` and `/confirm` with 10-minute TTL

Approval or confirmation commands are validated only against Redis records scoped to `userId + chatId` and consumed atomically when used.

## Extending to Other Platforms

The role system is platform-agnostic. Every tool definition declares its role tier. When you add GCP, Kubernetes, Terraform, or anything else — you assign the role on the tool. Everything else is automatic.

```typescript
// Tool declares its own role
const gcloudDeleteTool = createTool({
  id: 'gcloud_delete',
  role: 'commander',
  description: 'Delete a GCP resource',
  ...
})

const terraformApplyTool = createTool({
  id: 'terraform_apply',
  role: 'operator',
  description: 'Apply Terraform configuration',
  ...
})

const kubectlGetTool = createTool({
  id: 'kubectl_get',
  role: 'observer',
  description: 'Get Kubernetes resources',
  ...
})
```

The same Observer / Operator / Commander roles, the same activation flow, the same approval format — for every cloud, every platform, every tool Helmsman ever adds.
